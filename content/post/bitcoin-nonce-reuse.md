# Cryptographic concept 

When the nonce is reused it become vulnerable and result in a trivial equation
to be solved.

$$
Key = ((r \times (s_1 - s_2))^{p - 2} \mod{p}) \times ((m_1 \times s_2) - (m_2 \times s_1)) \mod{p}
$$

Where:

- $p$ Is a constant value defined by the $n$ value of [secp256k1](https://en.bitcoin.it/wiki/Secp256k1): $115792089237316195423570985008687907852837564279074904382605163141518161494337$
- $r$ and $s$ which are the [ECDSA signature pair](https://en.wikipedia.org/wiki/Elliptic_Curve_Digital_Signature_Algorithm)
- $m$ is the message to be signed, in this scenario, is the value that [*OP_CHECKSIG* will verify](https://en.bitcoin.it/wiki/OP_CHECKSIG)

Translating this equation as *Java* code it looks like:


```java
String pValue = "FFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFEBAAEDCE6AF48A03BBFD25E8CD0364141";
BigInteger p = new BigInteger(pValue, 16);
BigInteger privateKey = 
r.multiply(
           s1.subtract(s2)
          ).modPow(
                   p.subtract(BigInteger.valueOf(2)), p
                  ).
          multiply(
                   m1.multiply(s2).subtract(m2.multiply(s1))
                  ).mod(p);
```

A similar vulnerability found on
[PS3](https://events.ccc.de/congress/2010/Fahrplan/attachments/1780_27c3_console_hacking_2010.pdf)
in 2010.

In 2013 a smart hacker knowing about this launched an attack against the Bitcoin
transactions. Due to a bug on Androidâ€™s PRNG that made it not being initialized
correctly, transactions with more than one input reused the same $k$ as the
random value for the signature, which can be detected by simply finding a
duplicate $r$ value in the signatures of a transaction.

The bug was found in
[`SHA1PRNG_SecureRandomImpl`](https://android.googlesource.com/platform/libcore/+/kitkat-release/luni/src/main/java/org/apache/harmony/security/provider/crypto/SHA1PRNG_SecureRandomImpl.java#294)
class when not properly initialized the `engineNextBytes` function in the
`SecureRandom` implementation in *Apache Harmony* 6.0M3, as used in the Java
Cryptography Architecture (JCA) on Android before 4.4 and other products, when
**no seed is provided** by the user, it uses an incorrect offset value which
makes it easier for attackers to defeat cryptographic protection mechanisms by
leveraging the resulting PRNG predictability.
